Resumo executivo

O projeto não usa Supabase Realtime. O consumo concentra-se no PostgreSQL via Prisma e no Storage (signed URLs e uploads). O frontend nunca importa @supabase/supabase-js.

CamadaUso do Supabase
Banco (PostgreSQL)Prisma via DATABASE_URL — todas as leituras/escritas da API
StorageSupabaseStorageService — upload, signed URLs, buckets
AuthJWT próprio da API — não usa Supabase Auth no frontend

Maiores riscos

  • Polling HTTP no ERP (useOfficeEvents, 60s) em todas as rotas autenticadas
  • Portal dashboard com listQuotes sem paginação + N signed URLs por anexo
  • Listagens ERP sem take (OS, financeiro, orçamentos)
  • Dashboard financeiro dispara 3× dashboardFinancial + findMany sem limite
  • N+1 em loops (funcionários, comissões, sync de orçamentos)
  • Cron global de manutenção preventiva sem filtro por organização

Conclusão

O projeto está bem posicionado quanto a Realtime (não usa) e React Query (sem polling automático). Os maiores custos vêm de:

  1. Polling HTTP como substituto de Realtime no ERP
  2. Endpoints agregadores pesados (portal dashboard, dashboard financeiro)
  3. Listagens sem paginação que escalam com o crescimento da oficina
  4. Fan-out de signed URLs em anexos
